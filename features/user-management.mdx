## Overview

BioTea uses React Context API to manage user state globally throughout the application. The `UserContext` provides a centralized way to store and access user data across all screens and components.

## UserContext Implementation

The UserContext is implemented in `components/UserContext.tsx:1` using TypeScript for type safety:

```typescript components/UserContext.tsx
import React, { createContext, useContext, useState, ReactNode } from "react";

interface UserContextType {
    user: any;
    setUser: React.Dispatch<React.SetStateAction<any>>;
}

const UserContext = createContext<UserContextType | undefined>(undefined);

export function UserProvider({ children }: { children: ReactNode }) {
    const [user, setUser] = useState<any>(null);

    return (
        <UserContext.Provider value={{ user, setUser }}>
            {children}
        </UserContext.Provider>
    );
}

export function useUser() {
    const context = useContext(UserContext);
    if (context === undefined) {
        throw new Error("useUser must be used within a UserProvider");
    }
    return context;
}
```

## Architecture

<Accordion title="Context Pattern">
The UserContext follows React's recommended Context pattern:

1. **Context Creation** - Creates a typed context with `UserContextType`
2. **Provider Component** - Wraps the app and maintains user state
3. **Custom Hook** - Provides safe access to the context with error handling
</Accordion>

## Setting Up the Provider

The `UserProvider` must wrap your entire application. In BioTea, this is configured in `app/_layout.tsx:2`:

```typescript app/_layout.tsx
import { Stack } from "expo-router";
import { UserProvider } from "../components/UserContext";
import { SecureStoreProvider } from "../components/SecureStoreContext";

export default function RootLayout() {
  return (
    <UserProvider>
      <SecureStoreProvider>
        <Stack
          screenOptions={{
            headerStyle: {
              backgroundColor: '#f4511e',
            },
            headerTintColor: '#fff',
            headerTitleStyle: {
              fontWeight: 'bold',
            },
          }}
        >
          <Stack.Screen name="index" options={{ title: 'BioTea App' }} />
          <Stack.Screen name="login" />
          <Stack.Screen name="register" />
          <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
        </Stack>
      </SecureStoreProvider>
    </UserProvider>
  );
}
```

<Warning>
The `UserProvider` must be at the root level to ensure all screens and components can access user state. Always maintain this hierarchy when modifying the layout.
</Warning>

- Runtime errors are caught early in development

## Integration with Authentication

The UserContext works in conjunction with authentication and secure storage:

1. **Login** - User data is set in UserContext after successful authentication
2. **Persistence** - Data is also saved to SecureStore for persistence
3. **App Launch** - User data can be loaded from SecureStore and restored to UserContext

```typescript
// On login (app/login.tsx:36)
setUser(response.data.data);
await saveSecureData('user', response.data.data);

// On app launch (would be in _layout.tsx or similar)
const storedUser = await loadSecureData('user');
if (storedUser) {
    setUser(storedUser);
}
```

## Best Practices

<Note>
**Recommendations for UserContext usage:**

1. Always check if `user` is `null` before accessing properties
2. Use optional chaining (`user?.nombres`) to safely access nested properties
3. Clear user data on logout to prevent stale data
4. Synchronize UserContext with SecureStore for consistency
5. Consider creating a typed `User` interface instead of using `any`
</Note>
